FoodAlert: jak działa CLI monitorujące oferty Foodsi

FoodAlert to aplikacja konsolowa w Node.js i TypeScript. Dla wielu lokalnych profili loguje się do Foodsi, pobiera aktualne oferty, zapisuje bieżący widok użytkownika w SQLite i wyprowadza alerty o zmianach dostępności.

Model mentalny

Najkrótszy opis działania

Aplikacja nie jest serwerem. To lokalny proces CLI, który utrzymuje sesję bazy, odpala jednorazowe lub cykliczne przebiegi i drukuje powiadomienia w konsoli.

Offer Snapshot

Bieżący widok ofert jednego użytkownika w jednym przebiegu watchera. Snapshot aktualizuje restauracje, oferty i user_offer_states, a potem wskazuje oferty, które zniknęły.

Alert Derivation

Czystsza część reguł biznesowych. Z faktów zmian oraz polityki restauracji użytkownika tworzy zdarzenia: new-offer, re-stocked, sold-out i stock-change.

Provider Identity

Stabilny klucz w formacie provider:externalId. Chroni repozytoria przed luźnym sklejaniem stringów i pozwala rozszerzać projekt o kolejnych providerów.

Najważniejsze rozdzielenie odpowiedzialności: Foodsi dostarcza surowe dane, mapper zamienia je na OfferInput, snapshot zapisuje stan, a Alert Derivation decyduje, czy warto powiadomić użytkownika.

Jak projekt jest napisany

Kod jest mały, ale ma jasny podział. Adaptery wejściowe nie powinny zawierać reguł alertów, a repozytoria nie powinny znać mechaniki powiadomień. WatcherService pełni rolę koordynatora i celowo jest cienki.

Baza działa jako pojedyncza aktywna Database Session. Start CLI inicjalizuje SQLite, uruchamia migracje z katalogu drizzle, włącza WAL i foreign keys, zasiewa domyślne ustawienia, a repozytoria pobierają uchwyt przez getDb().

TypeScript działa w trybie strict, moduły są ESM (module: nodenext), a testy są uruchamiane przez Vitest bez równoległości plików, co pasuje do globalnej sesji bazy w testach.

Dane

Model bazy danych

Baza rozdziela globalną wiedzę o restauracjach i ofertach od widoku konkretnego użytkownika. To pozwala wielu profilom patrzeć na te same oferty, ale mieć własny stan i własne listy restauracji.

users profil

  • name, foodsiEmail, foodsiPassword
  • notifyOnlyFavorites
  • watchIntervalMinutes

app_settings globalne

  • id = 1
  • defaultWatchIntervalMinutes

restaurants provider

  • provider + externalId jako unikalny klucz
  • name, logoUrl, address
  • lastSeenAt, createdAt, updatedAt

offers provider

  • provider + externalId jako unikalny klucz
  • restaurantId jako FK do restaurants
  • currentQuantity i szczegóły odbioru

user_offer_states per user

  • userId + offerId jako unikalny klucz
  • currentQuantity użytkownika
  • lastSeenAt

favorites / ignored listy

  • userId + restaurantId
  • oddzielne tabele dla favorites i ignored
  • createdAt
Element Rola Ważna konsekwencja
offers.currentQuantity Ostatnio znana ilość oferty jako byt globalny. Może być aktualizowana przez fetch dowolnego użytkownika.
user_offer_states.currentQuantity Bieżący stan oferty widziany przez konkretnego użytkownika. To ta wartość jest używana do porównania w Offer Snapshot.
restaurants.lastSeenAt Pomaga utrzymać listę ostatnio widzianych restauracji. CLI może dodawać restauracje do favorites/ignored także spoza aktualnych ofert.
Migracje vs schema Kod schematu deklaruje cascade dla tabel użytkownika. Pierwsza migracja nie zawiera cascade, więc deleteUser czyści dane ręcznie.
Alert Derivation

Reguły zamiany zmian na powiadomienia

Reguły są deterministyczne i testowane jako czysta funkcja. Dane polityki restauracji są pobierane osobno przez deriveAlertEventsForUser.

Fakt ze snapshotu Warunek Alert
Oferta aktualna currentQuantity === previousQuantity Brak alertu.
Oferta aktualna Poprzednio 0, teraz > 0, oferta nie istniała wcześniej. new-offer
Oferta aktualna Poprzednio 0, teraz > 0, oferta istniała wcześniej. re-stocked
Oferta aktualna Poprzednio > 0, teraz 0. sold-out
Oferta aktualna Ilość zmieniła się, ale nie pasuje do powyższych przypadków. stock-change
Oferta zniknęła Była w poprzednim stanie użytkownika, nie ma jej w aktualnym fetchu. sold-out z ilością 0.

Ignored ma pierwszeństwo

Jeśli restauracja jest ignorowana, alert nie powstaje nawet wtedy, gdy jest też ulubiona.

Favorite-only zawęża

Gdy notifyOnlyFavorites jest włączone, alerty przechodzą tylko dla ulubionych restauracji.

Notifier jest końcówką

ConsoleNotifier tylko prezentuje gotowy AlertEvent. Nie decyduje o regułach.

Jakość

Co pokrywają testy

Testy są mocno domenowe. Największa wartość jest w testach snapshotu, watchera i alertów, bo one zabezpieczają reguły, które użytkownik realnie odczuwa jako powiadomienia.

Snapshot i watcher

Testy sprawdzają batch upsert, reuse restauracji, aktualizację ilości, zniknięte oferty jako sold-out oraz rollback transakcji, gdy jedna oferta jest niepoprawna.

Alert Derivation

Pokryte są wszystkie typy alertów oraz filtrowanie przez ignored i favorite-only.

Adaptery i workflowy

Mapper Foodsi ma testy fallbacków i parsowania, FoodsiClient test wyjaśnienia DNS, a CLI Workflows testują routing mutacji i usuwanie profilu.

Obszar Pliki testów Wniosek
Domena stanu offerSnapshot.test.ts, alertDerivation.test.ts Najważniejsze reguły są utrwalone w testach jednostkowych i integracyjnych z SQLite.
Runtime watcherService.test.ts, schedulerService.test.ts Watcher ma testy ścieżki transakcyjnej, scheduler timing ma testy jittera.
Granice systemu foodsiClient.test.ts, mapFoodsiOffer.test.ts Mapowanie API jest dobrze doprecyzowane, sam klient Foodsi ma na razie wąski test błędu DNS.

Ograniczenia

Miejsca, które warto znać przed zmianami

To nie są błędy blokujące działanie, tylko świadome koszty obecnego kształtu aplikacji i dobre punkty startowe do kolejnych iteracji.

Hasła plaintext w SQLite

README wprost wskazuje to jako znane ograniczenie. To najważniejszy temat bezpieczeństwa.

Spoofowane nagłówki Androida

FOODSI_HEADERS mogą wymagać aktualizacji, gdy Foodsi zmieni aplikację mobilną lub wymagania API.

Brak retry i backoff

Pojedynczy błąd sieci kończy dany przebieg użytkownika. Scheduler spróbuje ponownie dopiero w następnym cyklu.

Logowanie per fetch

Tokeny Foodsi nie są cache'owane. To upraszcza kod, ale zwiększa liczbę requestów auth.

Globalna aktywna sesja DB

Repozytoria zależą od getDb(). Dla CLI jest to proste, ale utrudnia równoległe testy i wiele baz naraz.

Tylko jeden provider

Typ Provider i klucze providerowe są gotowe na rozszerzenie, ale orkiestracja pobiera dziś tylko Foodsi.

Jednozdaniowa mapa przepływu

CLI wybiera użytkownika, WatcherService pobiera OfferInput[] z Foodsi, recordOfferSnapshot zapisuje różnice w SQLite, deriveAlertEventsForUser filtruje je przez favorites i ignored, a ConsoleNotifier drukuje wynik.

Najważniejszy punkt kontroli jakości to pilnowanie, żeby reguły Alert Derivation nie wyciekały do CLI, notyfikatora ani repozytoriów. Dzięki temu zmiana sposobu powiadamiania nie powinna zmieniać sposobu wykrywania zdarzeń.
